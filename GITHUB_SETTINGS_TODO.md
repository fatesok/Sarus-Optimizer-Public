# GitHub Settings To Do Manually

Some popularity and discoverability settings must be changed from the GitHub web interface. This checklist is for repository owner actions.

## 1. Add repository description

Suggested description:

> Public roadmap and benchmark summaries for Sarus-Optimizer: an explainable facility, transport, capacity, and ROI optimization system.

Path:

```text
Repository page -> About section -> gear/edit icon -> Description
```

## 2. Add repository topics

Suggested topics:

```text
facility-planning
optimization
location-optimization
capacity-planning
urban-planning
transport-planning
infrastructure
stadium-planning
airport-planning
matlab
decision-support
spatial-analysis
roi-analysis
scenario-analysis
```

Path:

```text
Repository page -> About section -> gear/edit icon -> Topics -> Save changes
```

## 3. Set website link

Suggested website URL:

```text
https://sarustesis.com
```

Path:

```text
Repository page -> About section -> gear/edit icon -> Website
```

## 4. Enable GitHub Pages

Recommended simple setup:

```text
Repository -> Settings -> Pages -> Build and deployment -> Deploy from a branch -> main -> /root -> Save
```

After GitHub Pages is enabled, GitHub will provide a public site URL.

## 5. Upload social preview image

A social preview image improves how the repository looks when shared on LinkedIn, X, Slack, and other platforms.

A design file has been added here:

```text
assets/sarus-optimizer-social-preview.svg
```

GitHub social preview upload may require PNG, JPG, or GIF. If SVG is not accepted, export or screenshot the SVG at approximately 1280 x 640 pixels and upload the PNG.

Path:

```text
Repository -> Settings -> Social preview -> Edit -> Upload image
```

## 6. Create the first GitHub Release

Suggested release tag:

```text
v0.1-public-foundation
```

Suggested release title:

```text
Sarus-Optimizer Public Foundation
```

Suggested release notes:

```text
Initial public documentation foundation for Sarus-Optimizer: vision, roadmap, milestones, use cases, benchmark-summary structure, public case narratives, FAQ, rights notice, and SVG public visuals. Source code remains private.
```

Path:

```text
Repository -> Releases -> Draft a new release
```

## 7. Enable Discussions or Issues for public feedback

Recommended first step: enable Issues only after you are ready to respond to public questions.

Potential public feedback labels:

```text
question
case-study
benchmark-summary
public-feedback
documentation
```
